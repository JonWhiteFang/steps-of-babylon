## Fix 1 — #236: premium-currency spend+grant are non-atomic

### Problem (code-grounded)
Two flows debit a permanent premium currency, then grant in a **separate** DB write — a
crash/cancel between them permanently loses the currency with nothing delivered:

- `OpenCardPack.kt:31` `spendGems(...)` → loop of `addCard`/`incrementCopyCount` (`:41-47`). Gems are real-money-purchasable.
- `UnlockUltimateWeapon.kt:29` `spendPowerStones(...)` → `:30` `uwRepository.unlockWeapon(type)`.

CLAUDE.md / CONSTRAINTS explicitly require currency spends to go through the **atomic
guarded-deduct pattern inside a `@Transaction`**. The #122 work made the *deduct* guarded
(gates grant on rows-affected) but left the deduct and the grant as two independent transactions.

### Design
All relevant tables (`player_profile`, `card_inventory`, `ultimate_weapon_state`) live in the
single `AppDatabase` (confirmed: `AppDatabase.kt` lists all 13 entities; every DAO is `db.xxxDao()`
off one `@Singleton`). Room's transaction tracker is **database-scoped, not DAO-scoped**, so a
`@Transaction` default method on one DAO that calls sibling DAOs shares one SQLite transaction —
exactly the `MilestoneDao.claimMilestoneAtomic` precedent (`MilestoneDao.kt:56-82`).

**Decision: keep the cross-DAO leak OUT of the domain layer.** `ClaimMilestone` already injects
DAOs into the domain layer (a tolerated leak), but issues #227/#228 actively flag that as debt.
We will NOT add a new domain→data leak. Instead:

- The new atomic `@Transaction` methods live on **`CardDao`** and **`UltimateWeaponDao`**, each
  taking `playerDao: PlayerProfileDao` as a parameter (mirroring `claimMilestoneAtomic`).
- They are exposed through the **existing repository ports** so the use cases keep depending only
  on `CardRepository` / `UltimateWeaponRepository` (no new domain imports).

#### CardDao.openCardPackAtomic
```kotlin
@Transaction
suspend fun openCardPackAtomic(
    gemCost: Long,                 // 0 ⇒ free pack: skip the deduct entirely
    cardTypeNames: List<String>,   // the 3 pre-rolled card type names (rolling stays in the use case — it's pure/seeded)
    playerDao: PlayerProfileDao,
): List<Boolean>? {               // null ⇒ deduct failed (insufficient); else per-card isNew flags
    if (gemCost > 0L) {
        if (playerDao.spendGemsAtomic(gemCost) == 0) return null   // guarded; rolls back nothing yet
    }
    return cardTypeNames.map { name ->
        val existing = getByType(name)
        if (existing != null) { incrementCopyCount(name); false }
        else { insert(CardInventoryEntity(cardType = name)); true }
    }
}
```
Rationale for passing pre-rolled names: rarity rolling is pure + seeded (`Random`), must stay in
the use case for testability (the `R35` guards depend on `pickCardType`). The DB write set (deduct
+ 3 card writes) is what must be atomic — and now is.

#### UltimateWeaponDao.unlockWeaponAtomic
```kotlin
@Transaction
suspend fun unlockWeaponAtomic(weaponType: String, powerStoneCost: Long, playerDao: PlayerProfileDao): Boolean {
    if (playerDao.spendPowerStonesAtomic(powerStoneCost) == 0) return false   // guarded deduct first
    val existing = getByType(weaponType)
    if (existing == null) upsert(UltimateWeaponStateEntity(weaponType = weaponType, isUnlocked = true))
    else markUnlocked(weaponType)
    return true
}
```

#### Repository ports
- `CardRepository`: add `suspend fun openCardPackAtomic(gemCost: Long, cardTypeNames: List<String>): List<Boolean>?`
  - Impl delegates to `dao.openCardPackAtomic(gemCost, cardTypeNames, playerDao)` — `CardRepositoryImpl`
    must now inject `PlayerProfileDao` (added to its `@Inject constructor`; Hilt provides it already).
- `UltimateWeaponRepository`: add `suspend fun unlockWeaponAtomic(type: UltimateWeaponType, powerStoneCost: Long): Boolean`
  - Impl delegates to `dao.unlockWeaponAtomic(type.name, powerStoneCost, playerDao)` — inject `PlayerProfileDao`.

#### Use-case rewrites
- `OpenCardPack.invoke`: roll the 3 rarities/types as today (pure), collect `cardTypeNames`, then a
  single `cardRepository.openCardPackAtomic(gemCost = if (isFree) 0 else packTier.gemCost, names)`.
  `null` → `Result.InsufficientGems`; else map names+isNew flags → `CardResult` (copiesAwarded stays 1).
  The empty `gems < gemCost` pre-check (`:28`) stays as a cheap fast-path (still returns
  `InsufficientGems`); the **authoritative** gate is the atomic deduct's null return.
- `UnlockUltimateWeapon.invoke`: keep the `owned.any{...isUnlocked}` + `powerStones < cost` pre-checks
  (cheap, preserve behaviour), then `return uwRepository.unlockWeaponAtomic(type, type.unlockCost.toLong())`.

### Tests (TDD, RED first)
- **Real-Room DAO tests** (Robolectric, mirror `GuardedClaimDaoTest`/`BillingReceiptDaoTest`):
  new `PremiumSpendDaoTest` (or extend `GuardedClaimDaoTest`):
  - `openCardPackAtomic` deducts gems + writes 3 cards in one transaction; returns isNew flags.
  - free pack (`gemCost=0`) writes cards, no deduct.
  - insufficient gems → returns `null`, **no card rows written, balance unchanged** (the atomicity proof).
  - `unlockWeaponAtomic` deducts power stones + sets `isUnlocked`; insufficient → `false`, no unlock, balance unchanged.
  - existing-row path: `markUnlocked` flips the flag (doesn't duplicate).
- **Use-case tests** (existing `OpenCardPackTest` / `UnlockUltimateWeaponTest` keep passing):
  update the fakes (`FakeCardRepository`, `FakeUltimateWeaponRepository`) to implement the new
  port methods, routing the deduct through a linked `FakePlayerRepository` (mirror `FakeMilestoneDao`'s
  `linkedPlayer`). The existing `R122 stale snapshot` tests must still pass (the atomic deduct is
  the guard now). Add a `…AtomicCallCount`-style assertion proving the atomic path is live.

### Fragile-zone note
This *extends* ADR-0020 (economy atomicity) to the two premium flows. Add an ADR-0020 amendment
or a new short ADR documenting the two new atomic methods. Do NOT remove the cheap pre-checks
(they preserve the `InsufficientGems`/`false` fast path without a DB round-trip when obviously broke).

## Fix 2 — #195: MissionsViewModel never re-subscribes on day-rollover

### Problem (code-grounded)
`MissionsViewModel.kt:57` `private var today` is captured once at `:84` inside
`combine(dailyMissionDao.getByDate(today), …)`. The 1 s ticker (`:69-80`) reassigns `today` but
the combined `StateFlow` is already subscribed to the original Flow → `getByDate` never
re-subscribes. After midnight the screen shows yesterday's missions. `HomeViewModel`/`StatsViewModel`
already solve this with `MutableStateFlow<date>` + `flatMapLatest` + an ON_RESUME observer.
`MissionsScreen` has no ON_RESUME refresh.

### Design (mirror HomeViewModel exactly)
- Replace `private var today = timeProvider.today().toString()` with
  `private val _today = MutableStateFlow(timeProvider.today().toString())`.
- Annotate the class `@OptIn(ExperimentalCoroutinesApi::class)`; import `flatMapLatest`.
- Wrap the `combine(...)` (`:83-107`) in `_today.flatMapLatest { today -> combine(getByDate(today), …){…} }`
  so `getByDate` re-subscribes when the date changes. `tick` stays inside `combine` for the midnight countdown.
- Add `fun refreshDate()` mirroring `HomeViewModel.refreshDate()`: read `timeProvider.today().toString()`;
  if changed, set `_today.value = now` AND `viewModelScope.launch { generateMissions(now); updateWalkingMissionProgress() }`.
- **Keep the existing 1 s ticker** (it drives the per-second midnight countdown — Missions shows
  "Resets in Xh Ym", unlike Home/Stats). On rollover the ticker now calls `refreshDate()` instead of
  mutating the var. `cancelForTest()` stays (the ticker still exists).
- Internal `today` reads in `claimMission` (`:113`) and `updateWalkingMissionProgress` (`:154-155`)
  read `_today.value`.
- `MissionsScreen.kt`: add the ON_RESUME `DisposableEffect` block (verbatim from `StatsScreen.kt:25-33`)
  calling `viewModel.refreshDate()`, plus the 4 lifecycle imports. (Belt-and-suspenders with the ticker;
  matches siblings and covers the resume-from-background case immediately.)

### Tests (TDD, RED first)
- New `MissionsViewModelTest` case: construct VM with `FakeTimeProvider(fixedDate = day1)`, collect
  `uiState` in `backgroundScope`, assert day1 missions; set `fakeTime.fixedDate = day2`, call
  `vm.refreshDate()`, advance, assert `uiState.value.missions` now reflects day2's generated rows.
  `FakeDailyMissionDao.getByDate` is StateFlow-backed so re-subscription re-emits. Call `cancelForTest()` last.

## Fix 3 — #193: no user signal when step-counter hardware is absent

### Problem (code-grounded)
All 3 sensor call sites silently no-op when `getDefaultSensor(TYPE_STEP_COUNTER)` is null
(`StepSensorDataSource.kt:28-33` closes the flow; `StepCounterService` foreground notification
starts regardless; `StepSyncWorker.kt:72` returns). No `hasSystemFeature` / `<uses-feature>` /
no-sensor UI state anywhere. On a no-sensor device the player earns nothing with no explanation.

### Design (minimal, well-scoped — surface it in onboarding)
- Add `fun isSensorAvailable(): Boolean = sensorManager.getDefaultSensor(Sensor.TYPE_STEP_COUNTER) != null`
  to `StepSensorDataSource` (it already owns `SensorManager` + the constant — single source of truth).
- `OnboardingViewModel` injects `StepSensorDataSource`; exposes `val stepSensorAvailable: Boolean = stepSensorDataSource.isSensorAvailable()`.
- `OnboardingScreen` gains a `stepSensorAvailable: Boolean` param. Add a **highest-priority** branch
  in the final-slide `when` (BEFORE `stepCountingGranted` — granting permission on a no-sensor device
  is meaningless): show an error-styled message ("This device has no step-counter sensor. You can
  still earn Steps via Health Connect — connect it in Settings.") + a `TextButton("Continue anyway")`
  → `finish()`. Reuse the error-text/TextButton pattern at `OnboardingScreen.kt:245-261`.
- `MainActivity` passes `stepSensorAvailable = onboardingViewModel...`? — the screen already uses
  `hiltViewModel()`, so read it inside the composable from the VM rather than threading through
  MainActivity (the VM is the natural owner; MainActivity's `OnboardingScreen(...)` call site does NOT
  need a new param). **Decision: expose via the VM, read in the composable** — keeps MainActivity
  untouched and matches how `slides` already flow VM→screen.

> NOTE: the in-app onboarding signal is the fix scope (issue: "expose the no-sensor state so the UI
> can message it"). A persistent Home banner + `<uses-feature required=false>` manifest entry are
> explicitly out of scope (deferred — issue says "at minimum, expose the no-sensor state").

### Tests (TDD, RED first)
- `OnboardingViewModelTest` (pure JVM, Mockito): `mock<StepSensorDataSource>()` directly — **NO
  interface needed.** mockito-core 5.x's default inline byte-buddy MockMaker mocks final classes
  (proven in-repo by `BillingManagerImplTest`/`RewardAdManagerImplTest` mocking `mock<Activity>()`, a
  concrete final class, with no `org.mockito.plugins.MockMaker` resource present). Amended after the
  adversarial review refuted the "Kotlin classes are final → need an interface" reasoning.
  Test: `whenever(sensor.isSensorAvailable()).thenReturn(false)` → assert `vm.stepSensorAvailable == false`,
  and the true case. (Open-verification item 1 below is resolved: mock the class.)
