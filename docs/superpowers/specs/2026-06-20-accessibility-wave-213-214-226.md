## #213 — Primary-button label contrast fails WCAG AA-normal (medium · small · confirmed)

**Confirmed.** `Theme.kt:16` sets `onPrimary = DeepBronze` (`#6B3A2A`) on `primary = Gold`
(`#D4A843`) → **4.19:1**. Material3 `Button` labels are `labelLarge` 14sp SemiBold = "normal" text, so
the **4.5:1** threshold applies and fails. ~30 primary CTAs inherit this. The audit also flags
`StatusDanger` (`#FF7043`, 3.38:1) and `RaritySand` (`#C2B280`, 4.41:1) on normal text.

**Fix.** Darken `onPrimary` to **`#4A2618`** (audit's recommended value → ~5.99:1 on Gold, passes AA
normal). One token edit in `Color.kt` + wire in `Theme.kt` (currently `Theme.kt:16` uses `DeepBronze`
directly — introduce a dedicated `OnGold` role token so the button-text colour is independent of the
brand `DeepBronze` fill).

- **`StatusDanger` / `RaritySand` — survey COMPLETE (F2), no change needed.** `StatusDanger` (#FF7043)
  is used only as Material `error` (`Theme.kt:21`) and as an **icon tint** (`CurrencyDashboardScreen.kt:197`,
  16dp `Close` icon) — never a normal-text foreground. `RaritySand` (#C2B280) is used only as a **fill**
  (`Rarity.kt` badge background + border/accent; the badge's text is dark `#1A1A2E` ON RaritySand, not
  RaritySand-as-text). Both pass the 3:1 non-text/large threshold and never gate normal text. **So
  `onPrimary`/`OnGold` is the SOLE #213 code change** — no brand-token change (consistent with the
  "don't touch a fill-only colour" rule).

**Test.** A pure JVM contrast guard: compute the WCAG contrast ratio of `OnGold` on `Gold` and assert
`>= 4.5`. WCAG relative-luminance math is pure (no Android — `Color` is an Android class, but the ARGB
ints are plain; compute luminance from the channel ints). New `ContrastTest` (mirrors the
`ColorLerpTest`/`RarityTest` pure-token-math precedent). Pin `onPrimary`/`OnGold` ≥ 4.5:1 so a future
token tweak that breaks it fails the build.

## #214 — Battle SurfaceView invisible to TalkBack (medium · medium · confirmed)

**Confirmed.** `GameSurfaceView` is a canvas — enemies/health/wave-outcome are pixels with no
accessibility nodes. A blind player gets zero combat feedback (rounds are *completable* — controls +
post-round overlay are labelled, engine auto-fires UWs — but there's no progress feedback).

**Fix (audit's recommendation — a polite Compose live region fed from `uiState`, NOT canvas a11y).**
Add an (visually-invisible) Compose node in `BattleScreen` with `liveRegion = LiveRegionMode.Polite`
and a `contentDescription` derived from the battle `uiState`, announcing **transitions** in:
- **Wave** (`currentWave`) — "Wave N".
- **Wave phase** (`wavePhase`) — "Spawning" / "Cooldown" (the existing phase string, humanised).
- **Ziggurat health bracket** — announce on crossing coarse brackets (e.g. each 25%: "Ziggurat health
  75%") rather than every HP tick, so TalkBack isn't spammed every frame (`currentHp`/`maxHp` update at
  the 200ms VM poll cadence — even that is too chatty for a live region; bucket it).
- **Round outcome** (`roundEndState` non-null) — "Round over. Wave N reached." and **battle error**
  (`battleError`) — "Battle stopped due to an error."

**Design details:**
- A pure helper / small `BattleAnnouncer` (holding the last announced bracket/wave/phase) computes the
  announcement and returns `null`/the same string when nothing changed — so the node's
  `contentDescription` only changes on a real transition (a live region announces on text *change*).
  This is the testable seam (pure, JVM).
- **Pinned signature + edge guards (F1 — REQUIRED).** `wavePhase` is the **raw enum name** — live
  values are `"SPAWNING"` / `"COOLDOWN"` / `""` (populated from `spawner?.phase?.name ?: ""`,
  `BattleViewModel.kt:232`), and `currentHp`/`maxHp` are `Double` with `maxHp` defaulting to `0.0`
  (`BattleUiState.kt:30-31`). So the helper signature is
  `battleAnnouncement(currentWave: Int, wavePhase: String, currentHp: Double, maxHp: Double,
  roundEndState: RoundEndState?, battleError: Boolean): String?` and MUST:
  - **return `null` when `wavePhase` is blank OR `maxHp <= 0.0`** (pre-round window) — no spurious
    "Wave 1 / Spawning" or NaN before the round is live;
  - **humanise the raw enum** (`SPAWNING` → "Spawning", `COOLDOWN` → "Cooldown");
  - **bucket health** as `floor(currentHp / maxHp * 4)` (25% brackets) and guard the divide so
    `maxHp <= 0.0` can't divide-by-zero / NaN.
- **Node shape (F5):** a **separate zero-size sibling** `Box(Modifier.size(0.dp).semantics {
  liveRegion = LiveRegionMode.Polite; contentDescription = ann })` — **NOT** `clearAndSetSemantics`
  (that clears descendants, the wrong tool for a leaf announcer). It must NOT draw anything visible and
  must NOT touch the existing steps-earned semantics node (`BattleScreen.kt:169`). `liveRegion` /
  `LiveRegionMode` is net-new in this repo but is the correct `androidx.compose.ui.semantics` API on the
  pinned Compose BOM.
- **Throttling:** health is bucketed (coarse brackets); wave/phase/outcome are naturally low-frequency.
  The announcement string only updates when a bucket/wave/phase/outcome boundary is crossed (the helper
  returns the same string otherwise, so no re-announce). Confirm the live region re-announces on string
  change only — if Compose re-announces on every recomposition with the same text, gate via
  `remember`/`derivedStateOf` so the semantics value is stable between transitions.
- All announcement text is localizable (`strings.xml`), matching the existing battle-string convention
  (`battle_wave_header`, etc.).
- This is presentation-only; the engine/VM are untouched (the VM already exposes everything via
  `BattleUiState`). It satisfies the GDD's "Battle screen provides audio cues" / TalkBack line for the
  combat surface (menus are already TalkBack-labelled).

**Test.** Pure JVM test of `battleAnnouncement`/`BattleAnnouncer`: feeds a sequence of states and
asserts (a) wave/phase/outcome transitions produce the right strings, (b) health announces only on
bracket crossings (not every HP tick), (c) no change → returns null/same (no spam). The Compose
live-region wiring itself is build/on-device-verified (documented boundary — the repo has no Compose UI
tests, #253; don't stand up a Robolectric Compose harness for this).

## #226 — GDD-promised color-blind palettes absent (medium · medium · partial → DEFER + non-color cues)

**Confirmed-partial.** GDD §17 (`StepsOfBabylon_GDD.md:459`) flatly promises "Color-blind modes: Three
alternative palettes for common color vision deficiencies" — none exist. Status is conveyed by colour +
small icons; the **battle wave-phase bar** (`BattleScreen.kt:156-161`) conveys phase by bar colour
(green SPAWNING / orange COOLDOWN) — though a phase *label* already sits directly above it
(`BattleScreen.kt:155`), so the bar is not strictly colour-*only* at the HUD level.

**Developer decision (2026-06-20): DEFER honestly + add non-color cues.** Do NOT build a palette toggle
this wave (that's its own wave-sized effort — re-theming layer + Settings toggle + persistence). Two
parts:

1. **GDD honesty edit.** Reword §17's color-blind line (`StepsOfBabylon_GDD.md:459`) from a flat promise
   to a **tracked post-v1.0 deferral** (e.g. "Color-blind modes (post-v1.0): three CVD-safe palettes are
   planned; not in v1.0"). The store listing (`docs/release/play-store-listing.md`) makes **no**
   accessibility claim (verified) — so **no store edit needed**. **Only the GDD line is a user-facing
   flat promise (F3):** `master-plan.md:43` mentions "color-blind modes (3 palettes)" but already marks
   Plan 24 deferred (`master-plan.md:147`) — it's internally honest, **leave it**. Archived
   `plan-24-accessibility.md` / `pre-claude-devdocs/*` / `archive/*` are historical — **do not edit**
   (CLAUDE.md). (The GDD is an editable internal design doc; it's the honesty risk the audit names.)
2. **Add a non-color channel to the worst color-only status.** Audit the status surfaces and ensure each
   colour-coded status also carries shape/text:
   - **Wave-phase bar** (`BattleScreen.kt:155-161`): the phase label above it already provides the text
     channel — confirm it's legible and, if the plan finds the bar reads as colour-only in practice,
     append the phase word into the bar's own `contentDescription` (also helps #214) or keep the label.
     Low-risk; likely already satisfied by the existing label — **confirm, don't over-engineer.**
   - Survey `CurrencyDisplay` (`presentation/ui/CurrencyDisplay.kt`) + status-token sites: currencies
     already pair a tint with an **icon + numeric label** (icon()/label()), so currency is not
     colour-only. `StatusSuccess`/`StatusWarning`/`StatusDanger` sites generally pair with text/icon —
     the plan confirms each and adds a text/shape cue only where colour is genuinely the sole channel.
   - **Scope discipline:** this is "ensure no status is colour-ALONE," not a redesign. The deliverable
     is (a) the GDD edit and (b) any genuinely colour-only site getting a non-colour cue. If the survey
     finds everything already pairs colour with text/icon, that's a valid outcome — record it.

**Test.** Mostly doc + survey; no new automated test required for the deferral. If a non-color cue is
added to a pure helper, unit-test it; otherwise build + on-device confirmation (documented boundary).
