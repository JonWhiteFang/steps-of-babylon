# Data-integrity wave — #237 / #238 / #248

**Date:** 2026-06-19
**Issues:** #237 (migration-chain guard), #238 (Keystore decrypt-fail over-broad wipe), #248 (DB closed under async work)
**Source:** `docs/reviews/2026-06-18-complete-app-review.md` (data-integrity dimension)
**Type:** defensive correctness — protect non-regenerable local progress (Steps gate behind real walking).
No schema change. No economy/engine-logic change.

---

## Problem statement

Three independent data-loss / data-corruption surfaces, all in the persistence layer:

- **#237 (confirmed, small).** Nothing asserts that `AppMigrations.ALL` forms a contiguous version chain
  topping out at `AppDatabase`'s `@Database(version=…)`. The worst migration failure mode — bump the
  version + commit the new schema JSON (passes the CI drift guard) but forget to register the `Migration`
  object — ships a guaranteed launch crash for every upgrading user, and nothing in CI catches it.
- **#238 (partial, medium).** `DatabaseKeyManager.getPassphrase` catches **any** `Exception` on decrypt and
  responds with irreversible deletion of the whole DB (`prefs.clear()` + `wipeDatabaseFile`). A *transient*
  Keystore fault (OEM Keystore daemon restart, low-memory, post-OS-update) thus destroys all player progress.
  There is no distinction between "key provably absent" (true device-restore — unrecoverable, wipe is
  correct) and "key present but decrypt threw" (transient — must NOT wipe).
- **#248 (partial, medium).** `DataDeletionManager.deleteAllData` calls `database.close()` immediately after
  `WorkManager.cancelAllWork()` and `stopService()`, both of which are **asynchronous**. An in-flight
  `StepSyncWorker` coroutine or the foreground sensor collector can still be mid-write when `close()` runs →
  "attempt to re-open an already-closed object" / `IllegalStateException` on a background thread.

## Goals

1. A JVM-runnable guard that fails the build if the migration chain is non-contiguous or its max target ≠ the
   live DB version.
2. `getPassphrase` wipes **only** when the Keystore alias is provably absent; transient decrypt failures fail
   loudly and are retried on next launch (no wipe).
3. `deleteAllData` awaits work cancellation (bounded) and stops the service before `close()`, closing the
   concrete close-race. Keep the existing `recreate()` + Room lazy-reopen self-heal (developer-chosen,
   smallest-change approach).

## Non-goals

- No process-restart / ProcessPhoenix mechanism for #248 (the "clearAllTables, no close" alternative was
  considered and **rejected** this session — it changes wipe semantics and needs a relaunch primitive).
- No retry-UI / error-screen for #238 transient failures beyond "don't wipe, rethrow so the next launch
  retries". Surfacing a user-facing decrypt-error screen is out of scope (the DB simply fails to open and
  Room/DI will surface the throw; we are removing the silent destructive path, not adding UI).
- No change to the migration floor (migrations cover **7→12**; pre-v7 is a separate documented concern, #258).

## Design

### #237 — migration-chain contiguity guard (new JVM test, no prod change)

**Key technical fact (verified):** `androidx.room.Database` has `@Retention(CLASS)`, NOT `RUNTIME`. So
`AppDatabase::class.java.getAnnotation(Database::class)` returns `null` at runtime — **annotation reflection
will not work**. Read the authoritative version from a built DB instead.

New `MigrationChainTest` (Robolectric, mirrors `RoomSchemaTest` setup — plain in-memory builder, no SQLCipher
factory needed; version is cipher-independent):

1. Build an in-memory `AppDatabase`; read the live version via
   `db.openHelper.readableDatabase.version` (== `PRAGMA user_version`, baked into Room's generated code).
2. Read `AppMigrations.ALL`, extract each `(startVersion, endVersion)` from the `Migration` base class.
3. Assert:
   - **Contiguity:** sorted by `startVersion`, each migration's `endVersion == next.startVersion`, and every
     step is `end == start + 1` (no gaps, no overlaps).
   - **Tops out at the live version:** `max(endVersion) == liveVersion`.
   - **Floor is the documented migration floor (7):** `min(startVersion) == 7` — pins the contract that
     migrations begin at v7 so a *removed* early migration is also caught. (Asserted as a named constant with
     a comment tying it to #258's pre-v7 note; bump deliberately if the floor ever moves.)

This catches the forgotten-registration case: bumping `@Database(version=13)` without adding `MIGRATION_12_13`
makes `max(endVersion)=12 ≠ liveVersion=13` → test fails before release.

### #238 — narrow the wipe to "alias provably absent"

> **Review amendment (critical).** The wipe-vs-rethrow *decision* MUST be a pure, JVM-testable seam — do
> NOT attempt to test it through `getPassphrase` under Robolectric. Verified: `getPassphrase`'s try-block
> calls `decrypt(...)` → `getOrCreateKeystoreKey()` → `KeyStore.getInstance("AndroidKeyStore")`, which throws
> `NoSuchAlgorithmException` under Robolectric **before** any cipher/`doFinal` on a seeded blob (the existing
> `DatabaseKeyManagerTest` docstring documents exactly this). So a "malformed blob forces a decrypt failure"
> premise is false (the catch fires for the wrong reason), and the alias-absent branch can't return a fresh
> passphrase either — the fall-through `encrypt()` re-enters the keystore and throws again. Therefore:
> extract `decideOnDecryptFailure(e, aliasExists): Decision` (sealed `Wipe`/`Rethrow`) as a pure internal
> function and unit-test THAT purely (mirroring `validateChain`). `getPassphrase` just calls it and acts on
> the result. The full `getPassphrase` integration stays on-device-smoke, as the file already documents.

Introduce an **alias-existence seam** so the destructive branch is unit-testable without a working
AndroidKeyStore (Robolectric can't shadow it — the file already documents this limitation):

```kotlin
// new internal seam, default delegates to the real KeyStore
internal var keystoreAliasExists: () -> Boolean = ::realKeystoreAliasExists

private fun realKeystoreAliasExists(): Boolean =
    try {
        KeyStore.getInstance("AndroidKeyStore").apply { load(null) }.containsAlias(KEYSTORE_ALIAS)
    } catch (e: Exception) {
        // Can't even open the keystore → treat as "present/uncertain", i.e. do NOT wipe.
        Log.w(TAG, "Keystore unavailable while checking alias presence", e)
        true
    }
```

The pure decision helper (this is the JVM-tested unit):

```kotlin
internal sealed interface DecryptFailureAction {
    object Wipe : DecryptFailureAction      // alias provably absent → device restore → safe to wipe
    object Rethrow : DecryptFailureAction   // alias present/uncertain → transient → preserve progress
}

internal fun decideOnDecryptFailure(aliasExists: Boolean): DecryptFailureAction =
    if (aliasExists) DecryptFailureAction.Rethrow else DecryptFailureAction.Wipe
```

`getPassphrase` catch block becomes (acts on the pure decision):

```kotlin
} catch (e: Exception) {
    when (decideOnDecryptFailure(keystoreAliasExists())) {
      DecryptFailureAction.Wipe -> { /* clear prefs + wipeDatabaseFile, fall through to regenerate */ }
      DecryptFailureAction.Rethrow -> throw e
    }
}

// (illustrative; original branch bodies below)
if (!keystoreAliasExists()) {
        // Alias genuinely gone (device restore to new hardware) — the on-disk DB is encrypted
        // with the now-unrecoverable passphrase. Wipe so Room rebuilds instead of crash-looping.
        Log.w(TAG, "Keystore alias absent — wiping stale key blob and unreadable DB file", e)
        prefs.edit().clear().apply()
        wipeDatabaseFile(context)
        // fall through to regenerate
    } else {
        // Alias present but decrypt threw → transient/recoverable Keystore fault. Do NOT destroy
        // progress; rethrow so DB open fails this launch and we retry next launch.
        Log.e(TAG, "Transient passphrase decryption failure (alias present) — NOT wiping", e)
        throw e
    }
}
```

**Behavioural contract:**
- alias absent → wipe + regenerate (today's behaviour, now correctly scoped) → returns a fresh passphrase.
- alias present + decrypt throws → rethrow, **no wipe** (DB open fails this launch, retries next launch).
- The fall-through after a wipe must still reach `generateRandomPassphrase()` (don't `return` early in the
  wipe branch).

**Uncertainty handling:** if the keystore can't even be *opened* to check `containsAlias`, default to "do not
wipe" (treat as present). Preferring data-preservation over a possibly-spurious wipe is the safer default for
non-regenerable currency.

### #248 — await cancellation + stop service before close

`deleteAllData` ordering becomes:

1. `WorkManager.getInstance(context).cancelAllWork()` → **block (bounded) on the result** before proceeding.
   `cancelAllWork()` returns an `androidx.work.Operation`; await `.result` — a
   `com.google.common.util.concurrent.ListenableFuture` (exported transitively by `androidx.work`, NOT via a
   `kotlinx-coroutines-guava` dep, which this project does NOT have). Use the **raw** `Future.get(timeout)`
   (`java.util.concurrent.Future`), NOT a coroutines `await()` extension. Timeout ≤ **2s** (verified:
   `SettingsViewModel.deleteAllData` calls this synchronously off the Compose click handler → **main thread**;
   the 5s ANR window means the bound must be well under it). On timeout/exception: log + proceed (the wipe
   must still complete — a missed in-flight write is wiped anyway).
2. `context.stopService(...)` (unchanged — best-effort; the service's own teardown nulls its collector).
3. `database.close()` — now strictly after the cancel barrier.
4. Steps 4–7 unchanged (delete file, clear prefs, delete keystore alias, `recreate()`).

`deleteAllData` runs on the main thread (called from `SettingsViewModel.deleteAllData` directly). The bounded
`.get(timeout)` keeps the worst case to `timeout` (a few seconds) rather than indefinite; this matches the
existing synchronous shape of the method (it already does blocking file/prefs IO).

**Note on residual race:** the foreground sensor collector write is not synchronously joinable from here
(`stopService` is async). This fix closes the **WorkManager** half of the race deterministically and narrows
the service half (the service stops promptly; Room's lazy reopen-after-close still backstops a late write).
The remaining service-collector window is documented, not eliminated — eliminating it fully needs the
process-restart approach we rejected this session. This is an honest partial: it removes the most-likely
(worker) crash path. _(Stated explicitly so the review/STATE don't overclaim.)_

## Test strategy

- **#237:** `MigrationChainTest` (Robolectric) — green now; would go red on an unregistered/ gapped migration.
  Also add a negative-style assertion in-test (build a deliberately-gapped array locally and assert the
  validation helper rejects it) OR factor the chain-validation into a pure helper
  `AppMigrations.validateChain(migrations, liveVersion, floor)` and unit-test it pure (preferred — pure JVM,
  no Robolectric for the logic; the Robolectric test only supplies the real `liveVersion`).
- **#238 (revised per review — pure-unit only):** test `decideOnDecryptFailure(aliasExists)` purely (no
  Robolectric, no AndroidKeyStore): `aliasExists=false → Wipe`, `aliasExists=true → Rethrow`. Separately, a
  small Robolectric test can assert `wipeDatabaseFile(context)` actually deletes the db/-shm/-wal files (the
  existing `internal` seam) — that's the only filesystem behaviour reachable in JVM. Do NOT assert the full
  `getPassphrase` round-trip under Robolectric (it throws at `KeyStore.getInstance` regardless — would pass
  for the wrong reason). **Test-isolation:** any test that overrides the `keystoreAliasExists` `var` MUST
  reset it in `@After` (the `object` is shared across the JVM/Robolectric runs).
- **#248:** extend `DataDeletionManagerTest`. The existing harness uses `Configuration.Builder().build()`
  (no `SynchronousExecutor`) — for a deterministic "awaited before close" assertion, switch the test's
  WorkManager config to `setExecutor(SynchronousExecutor())` so `cancelAllWork().result` resolves on the
  calling thread. Assert: enqueue a worker → `deleteAllData` → DB closed without throwing + work cancelled.
  Existing 5 tests must stay green (they enqueue no work, so an empty-queue cancel resolves fast / times out
  harmlessly either way).

## Risks

- #238 rethrow path: a *persistent* (non-transient) decrypt failure with the alias still present would now
  crash-loop on launch instead of self-healing via wipe. This is the deliberate trade: we will not silently
  destroy non-regenerable progress on a fault we can't prove is unrecoverable. Mitigation: the alias-absent
  branch still auto-heals the true device-restore case (the common one). Persistent-present is rare and the
  user keeps the option of "Delete All Data" in Settings.
- #248 bounded wait on the main thread: keep the timeout small (≤ a few seconds) so the UI can't ANR.
