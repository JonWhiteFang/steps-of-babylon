- `Screen.items by lazy` + `argumentFreeRoutes by lazy` — both guard against sealed-class init-order NPE (commit 1872af9).
- `release/` directory — all gitignored. `release/upload-keystore.jks` is irreplaceable; losing it before Play App Signing enrollment = no app updates ever (already enrolled, so this is now mostly historical).
- **Live-price wiring (PR B)** — stable. Don't add price refresh on app resume / locale change without re-deriving the cache invalidation rules; the current "fetch once on Store entry" is intentionally simple for v1.
- **GOLDEN_ZIGGURAT × overdrive `fortuneMultiplier` stacking (RO-09 #2)** — the 3-site invariant ("higher buff wins; lower restores cleanly when one ends") is regression-guarded by 4 GameEngineTest entries. Don't add a fifth fortune source without extending those tests.
- **`GameEngine.entities` thread-safety (#118, PR #129)** — every structural mutation/iteration of `entities` is now confined behind the private `entitiesLock` monitor (`update`/`init`/`applyStats`-orb-reconcile/`render`-snapshot). Any NEW `entities` mutation must take the same lock or be confined to the loop thread; guarded by `GameEngineConcurrencyTest`.
- **GOLDEN damage layer (#119, PR #130)** — GOLDEN is now a re-derived multiplier (`goldenDamageMult`), not a stat snapshot; `updateZigguratStats` re-captures `preGoldenStats` while GOLDEN is active. Don't restore the snapshot-and-overwrite pattern.
- **Economy spend/claim contract (#122, PR #131, ADR-0020)** — `spendGems`/`spendPowerStones`/`spendStepsIfSufficient` return Boolean; **gate the grant on the result**. `spendSteps` keeps the clamp for the anti-cheat escrow only. One-shot claims use guarded `… AND claimed=0` + mark-first.
- **`DailyStepManager` Mutex (#120, PR #132)** — the credit read-check-write runs under a non-reentrant `Mutex` (`ensureInitializedLocked` is the in-lock init split). Don't call the locked methods from inside the lock or add an un-locked counter mutation.
- **`GameEngine.getAliveEnemies()` must NOT be cached across a frame (#125)** — it is re-derived live on every call on purpose. `EnemyEntity.takeDamage` re-fires `onDeath` on an already-dead enemy, and BLACK_HOLE/POISON_SWAMP both kill enemies mid-frame; a shared/stale alive-list snapshot would let one ongoing UW re-hit another's corpses and double-credit the kill. The #125 optimisation only de-duplicated the *allocation* (single-pass build), not the call. Guarded by the `R125` GameEngineTest entries — don't "finish" it into a per-frame cache.
- **Game-loop frame clamp (#126)** — `GameLoopThread` bounds the fixed-timestep catch-up backlog via `SimulationMath.clampAccumulator` (`MAX_CATCHUP_TICKS = 8`). The ceiling is tuned to engage only on a genuine multi-frame stall (a 30 fps render at 4× legitimately needs ~7.9 ticks/frame); don't lower it below ~8 without re-deriving against the worst legitimate slow-render case. Guarded by `SimulationMathTest.clampAccumulator*`.
- **`daily_step_record` writers must stay column-targeted (#121)** — the four per-field daily-step writers go through the DAO's column-targeted upserts (`setSensorAndCreditedSteps` / `setHealthConnectSteps` / `setActivityMinutes` / `setEscrow`: `INSERT … ON CONFLICT(date) DO UPDATE SET <only-its-columns>`), NOT a read-copy-`@Upsert` of the whole row. That disjoint-column property is what stops the concurrent sensor/worker/HC writers from clobbering each other. Don't reintroduce a whole-row `existing.copy(field=…)` upsert for these columns, and any NEW per-field writer must follow the same template. Guarded by `DailyStepDaoTest` (incl. the lost-update repro) + `StepRepositoryImplTest` delegation checks.
