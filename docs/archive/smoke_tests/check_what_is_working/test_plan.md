# Smoke Test Plan — 5 Areas × 5 Cases

Every smoke case references an artifact that **already exists** in the
repository. No new test files, no new mocks, no new harness. The plan pairs
"what we're smoking" (behaviour/surface) with "how we smoke it" (existing
test class and `@Test` method, or Gradle task). Tools used are the project's
existing JUnit 5 + Robolectric + Mockito-Kotlin + room-testing stack.

Legend:
- **File** — path under `app/src/test/java/com/whitefang/stepsofbabylon/` or a top-level build artifact.
- **Framework** — `JUnit5` (Jupiter), `JUnit4` (used by Robolectric runner in a handful of files), `Gradle` (build/lint task).
- **Real vs fake** — whether the case runs against real components or uses an existing in-memory fake / stub.

---

## Area 1 — Build & Packaging (no mocks, full pipeline)

Purpose: confirm the code compiles, annotation processors (KSP for Hilt +
Room) run clean, and the debug APK actually packages. These are the
cheapest, broadest failure signals.

| # | Case | How | File / task | Framework | Real vs fake |
|---|---|---|---|---|---|
| 1.1 | Kotlin production source compiles | `./run-gradle.sh compileDebugKotlin` | Task `:app:compileDebugKotlin` | Gradle | Real |
| 1.2 | Kotlin test source compiles | `./run-gradle.sh compileDebugUnitTestKotlin` | Task `:app:compileDebugUnitTestKotlin` | Gradle | Real |
| 1.3 | KSP runs (Hilt + Room codegen) | Implicit in 1.1; verified by presence of generated Hilt `_HiltComponents`, Room `*_Impl.kt` under `app/build/generated/ksp/debug/` | Task `:app:kspDebugKotlin` | Gradle | Real |
| 1.4 | Room schema is exported for current version | Check `app/schemas/com.whitefang.stepsofbabylon.data.local.AppDatabase/8.json` exists and is current (version 8 per `AppDatabase.kt`). Regenerated by any `assembleDebug` run. | Task `:app:assembleDebug` + filesystem check | Gradle | Real |
| 1.5 | Debug APK packages end-to-end | `./run-gradle.sh assembleDebug` produces `app/build/outputs/apk/debug/app-debug.apk` | Task `:app:assembleDebug` | Gradle | Real |

## Area 4 — Persistence Round-Trip (real Room via Robolectric)

Purpose: prove the Room schema (v8), TypeConverters, escrow lifecycle queries,
and SharedPreferences wrappers survive a round-trip. All cases use
`Room.inMemoryDatabaseBuilder(...)` or Robolectric `SharedPreferences`; no
emulator, no SQLCipher passphrase, no file I/O.

| # | Case | File | Framework | Real vs fake |
|---|---|---|---|---|
| 4.1 | Player profile round-trip (gems, power stones, current tier) | `data/local/RoomSchemaTest.kt` → `player profile round-trip` | JUnit4 + Robolectric | Real Room (in-memory) |
| 4.2 | Daily-step record round-trip including escrow fields (`escrowSteps`, `escrowSyncCount`) | `data/local/RoomSchemaTest.kt` → `daily step record round-trip with escrow fields` | JUnit4 + Robolectric | Real Room (in-memory) |
| 4.3 | End-to-end escrow lifecycle via `StepRepositoryImpl` (release + discard) | `data/integration/EscrowLifecycleTest.kt` | JUnit4 + Robolectric | Real Room + real repo |
| 4.4 | Widget provider SharedPreferences round-trip (balance render surface) | `service/StepWidgetProviderTest.kt` | JUnit4 + Robolectric | Real `SharedPreferences` |
| 4.5 | `StepIngestionPreferences` heartbeat + day-start counter round-trip over day rollover | `data/sensor/StepIngestionPreferencesTest.kt` | JUnit5 + Robolectric | Real `SharedPreferences` |

Note: Cases 4.1–4.4 use JUnit 4 annotations because they run under
`@RunWith(RobolectricTestRunner::class)`. They execute under the same Gradle
task as JUnit 5 tests because the Gradle build is configured with
`useJUnitPlatform()` and the Jupiter Vintage bridge / Robolectric's JUnit 4
runner path. If these turn out not to be discovered in a fresh environment,
that is captured as a risk in `report.md`.

## Smoke-run order (easiest first)

1. Area 2 — pure-Kotlin JVM tests, fastest feedback, zero environment setup.
2. Area 5 — JUnit 5 ViewModel tests, fakes only.
3. Area 3 — JUnit 5 with a couple of Mockito spies and Robolectric for prefs.
4. Area 4 — Robolectric-backed Room + SharedPreferences; slower first run.
5. Area 1 — build & packaging; slowest because it exercises KSP + AGP.

`report.md` follows this order when documenting the run.
